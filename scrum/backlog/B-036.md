# B-036 - File-relative schema reference resolution

- Status: done
- PO Priority: 1 (critical)
- Risk: 2 (medium)
- Owner: gresendesa
- Creation date: 2026-06-13
- Last updated: 2026-06-13
- Dependencies: B-033, B-035

## Description

Resolve the schema reference ambiguity introduced by local per-section schema
validation and file-scoped validation. A `schema` metadata value must always be
resolved relative to the Markdown file that contains the section declaring that
schema, regardless of whether validation runs in repository mode or file mode.

## Scope

- Replace repository-root-relative schema resolution in `mdb validate --root`.
- Preserve file-parent-relative schema resolution in `mdb validate --file`.
- Use the section's own `file_path` as the base for local schema references.
- Keep web URI schemas unsupported until B-034.
- Preserve `mdb validate --json` output shape and additive schema error fields.
- Update README, specification, llms guidance, architecture memory, and
  decisions.

## Out of scope

- Adding a global `--schema-dir` option.
- Defining a schema alias syntax such as `@schemas/...`.
- Fetching or resolving web URI schemas.
- Requiring all sections to declare schemas.

## Criteria of acceptance

- [x] A section in the repository root can reference a local schema using a path
      relative to its own Markdown file.
- [x] A section in a nested directory can reference `schema/foo.schema.json`
      and resolve it from that nested directory in `mdb validate --root`.
- [x] `mdb validate --file` continues to resolve schemas from the selected
      file's directory.
- [x] Documentation states that schema references are always file-relative.
- [x] Decisions record DEC-002 as the active schema resolution contract and
      preserve DEC-001 as replaced history.
- [x] Automated tests cover the repository-mode nested file regression.

## Risk assessment

- Implementation risk: medium, because repository validation may contain
  sections from multiple directories and each section needs its own schema base.
- Contract risk: medium, because nested files that relied on repository-root
  schema paths must update their metadata references.
- Documentation risk: low, because the new rule is simpler than the previous
  mode-dependent behavior.

## Notes

- The PO explicitly selected file-relative schema resolution as the invariant.
- `mdb validate` should not receive a global schema directory for normal
  relative schema references, because that would introduce a second source of
  truth.
