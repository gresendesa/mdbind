# B-032 - `mdb metadata` commands for structured YAML metadata

- Status: done
- PO Priority: 2 (high)
- Risk: 2 (medium)
- Owner: gresendesa
- Creation date: 2026-06-12
- Last updated: 2026-06-12
- Dependencies: B-013

## Description

Add metadata operations responsible for reading, changing, adding, and removing
key-value data inside the structured YAML metadata block associated with a
section id.

The operation must be restricted to the structured block. It must not edit
Markdown body content or unstructured text outside the YAML metadata block.

Input values are always provided as JSON, while persisted metadata remains YAML.
The caller must be able to address attributes and nested attributes at any depth
using dotted notation. The operation must also allow replacing an attribute that
already contains nested attributes in a single request.

The command family should be `mdb metadata`, with the following subcommands:

- `mdb metadata get` reads metadata from the structured YAML block.
- `mdb metadata update` adds or changes metadata, aligned with Python
  dictionary update semantics.
- `mdb metadata unset` removes metadata from the structured YAML block.

The names are intentionally explicit about the metadata target, so callers do
not confuse these operations with Markdown body editing.

## Scope

- Add the `mdb metadata get` user-facing operation to read values from the
  structured YAML metadata block for a target section.
- Add the `mdb metadata update` user-facing operation to update the structured
  YAML metadata block for a target section.
- Add the `mdb metadata unset` user-facing operation to remove values from the
  structured YAML metadata block for a target section.
- Resolve the target section by its section id.
- Accept JSON input for values written through `metadata update`.
- Return JSON output for `metadata get` by default when requested with the
  existing CLI JSON conventions.
- Persist the updated metadata as YAML in the section structured block.
- Support dotted paths such as `status`, `owner.name`, and
  `review.checklist.manual`.
- Create intermediate mappings when adding a nested dotted path, when safe and
  unambiguous.
- Allow replacing a whole attribute value, including objects that already contain
  nested attributes.
- Support removing top-level and nested metadata keys through dotted paths.
- Restrict writes to the structured YAML block only.
- Preserve Markdown body content and directives outside the structured block.
- Preserve existing contracts unless a versioned contract change is explicitly
  approved.
- Update `specification.md`, `llms.txt`, and `README.md` with the operation
  contract and examples.

## Out of scope

- Editing Markdown body content.
- Editing directive syntax outside the structured metadata block.
- Supporting non-JSON input formats for the requested value.
- Editing arbitrary YAML outside the section structured metadata block.
- Introducing a breaking change to the parser, index, or CLI JSON output
  contracts without versioning.

## Criteria of acceptance

- [x] `mdb metadata get` can return the full metadata object for a section
      selected by id.
- [x] `mdb metadata get` can return a top-level or nested value by dotted path.
- [x] `mdb metadata update` can add a top-level metadata key to the YAML
      structured block of a section selected by id.
- [x] `mdb metadata update` can update an existing top-level metadata key.
- [x] `mdb metadata update` can add or update nested metadata using dotted
      notation at any supported depth.
- [x] `mdb metadata update` can replace an attribute that currently contains
      nested attributes with a new JSON value in a single request.
- [x] Acceptance tests cover changing a flat scalar attribute value.
- [x] Acceptance tests cover changing a nested attribute value by dotted path.
- [x] Acceptance tests cover changing a value that is itself a nested JSON
      object, persisted back as a YAML mapping.
- [x] `mdb metadata unset` can remove a top-level metadata key.
- [x] `mdb metadata unset` can remove a nested metadata key by dotted path.
- [x] `mdb metadata unset` returns a clear error when the target path does not
      exist.
- [x] JSON object and array inputs are persisted as YAML mappings and sequences.
- [x] Write operations do not modify Markdown body content outside the
      structured block.
- [x] All metadata commands return a clear error when the target section id does
      not exist.
- [x] Metadata commands return a clear error when a dotted path cannot be
      applied because an intermediate value is not a mapping.
- [x] Automated tests cover full reads, dotted reads, top-level updates, nested
      dotted updates, flat scalar changes, nested scalar changes, nested object
      value changes, object replacement, top-level removal, nested removal,
      JSON-to-YAML persistence, and protected body content.
- [x] `specification.md` explains the `metadata get`, `metadata update`, and
      `metadata unset` contracts, JSON input contract, dotted path semantics,
      and YAML storage behavior.
- [x] `llms.txt` includes agent-oriented guidance for using the `mdb metadata`
      command family.
- [x] `README.md` includes user-facing examples for reads, top-level updates,
      nested dotted updates, object replacement, and removals.

## Dependencies

- B-013 provides the structured YAML metadata block syntax and parsing baseline.

## Risk assessment

- Implementation risk: medium, because write operations must modify source
  Markdown while preserving content outside the structured YAML block.
- Contract risk: medium, because read, update, and unset semantics for dotted
  paths and object replacement must be explicit and stable.
- Test risk: medium, because regression coverage must prove that only the
  structured block changes.

## Notes

- The PO described the operation as adding or changing key-value data in the
  YAML block where the section id lives.
- Dotted notation is intended for attributes and subattributes at any depth.
- Input is JSON by contract, but stored representation remains YAML.
- The repository currently uses `llms.txt` as the LLM guidance file.
- `metadata get` and `metadata unset` were added to the same sprint by PO
  request so the metadata command family ships coherently.
