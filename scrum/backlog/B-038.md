## Summary

When running `mdb validate --file <file>`, MDBind validates only the selected
Markdown file, but still reports `@ref` and `@include` targets in other files as
`broken_ref` or `broken_include`.

This is surprising because relative paths appear to resolve to the correct URI,
but referenced files are not indexed in `--file` mode. As a result, valid
project references can appear broken when validating a single file.

## Example

Given `scrum/architecture.md` containing references such as:

```markdown
[@ref: Technical specification](../specification.md#specification)
[@ref: Architecture decision](decisions.md#decisions.DEC-005)
[@ref: Template foundation](backlog/B-002.md#backlog.item.B-002)
```

Running:

```bash
mdb validate --file scrum/architecture.md
```

can produce errors like:

```text
ERROR [broken_ref] ... scrum/architecture.md#architecture:
target '/path/to/repo/specification.md#specification' not found in index

ERROR [broken_ref] ... scrum/architecture.md#architecture.implementation-baseline:
target '/path/to/repo/scrum/backlog/B-002.md#backlog.item.B-002' not found in index
```

However, full repository validation can succeed because the referenced files are
indexed:

```bash
mdb validate --root .
```

## Expected behavior

One of these would be less surprising:

1. `--file` validates only parsing, local section structure, and schemas, but
   does not report external refs as broken.
2. `--file` loads referenced files into the validation index when refs/includes
   point outside the selected file.
3. `--file` has an option such as `--resolve-refs` or `--root <repo>` to
   validate external refs against the repository graph.
4. External refs are reported with a distinct warning or error type, such as
   `external_ref_not_indexed`, instead of `broken_ref`.

## Actual behavior

`--file` reports valid external references as `broken_ref` because the target
files are outside the single-file index.

## Why this matters

For editor integrations and AI agents, `mdb validate --file` is useful as a fast
check after editing one Markdown file. Currently it can produce false-positive
broken refs for normal cross-file references, making it hard to distinguish real
broken references from references that were simply not indexed.

## Related observation

Schema resolution now works relative to the file declaring the schema, which is
useful for single-file validation. Ref/include validation needs similarly clear
semantics in `--file` mode.

## Scope

- Define the intended `--file` ref/include validation semantics.
- Preserve fast isolated validation for editor and agent workflows.
- Avoid reporting valid external references as ordinary `broken_ref` or
  `broken_include`.
- Preserve existing `mdb validate --root` behavior.
- Preserve schema validation behavior from B-036.
- Update README, specification, llms guidance, and tests.

## Out of scope

- Implementing web URI schema references from B-034.
- Changing directive syntax.
- Removing repository-wide broken reference validation.
- Requiring `--file` users to validate the whole repository by default.

## Risk assessment

- Implementation risk: medium, because validation currently uses a single-file
  graph in `--file` mode and broken reference checks are shared with root mode.
- Contract risk: medium, because changing error behavior may affect consumers

- UX risk: low to medium, because the change should reduce false positives but
  must keep real local broken refs visible.
