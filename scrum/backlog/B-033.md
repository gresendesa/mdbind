# B-033 - Local schema validation for structured section metadata

- Status: done
- PO Priority: 2 (high)
- Risk: 3 (high)
- Owner: gresendesa
- Creation date: 2026-06-12
- Last updated: 2026-06-12
- Dependencies: B-013, B-015, B-032

## Description

Add deterministic schema validation for each section YAML metadata block.
Sections that opt into schema validation must include a `schema` attribute in
their structured YAML metadata block. The value of `schema` points to the schema
used to validate that section metadata.

Schemas must be local-first and stored locally under `scrum/schema/` for this
item. The validation layer should support JSON Schema as the baseline schema
language, with YAML schema files accepted when they contain the same JSON Schema
document serialized as YAML.

This feature adds another deterministic contract layer on top of the existing
section metadata model.

## Scope

- Add schema validation for structured section metadata blocks that declare a
  `schema` attribute.
- Keep schema validation per-section only. There is no global schema for the
  repository in this item.
- Support local schema references, especially schemas stored under
  `scrum/schema/`.
- Defer web URI schema references to B-034.
- Treat JSON Schema as the normative validation language.
- Accept YAML schema files when they encode a JSON Schema document.
- Validate the full section metadata object, including custom attributes.
- Preserve the existing `section` identifier contract.
- Report deterministic validation errors with section URI, schema reference,
  metadata path, and human-readable detail.
- Integrate schema checks into `mdb validate`.
- Preserve existing `mdb validate --json` contract through additive error
  fields such as `schema`, `path`, and `schema_path`.
- Update `README.md`, `specification.md`, and `llms.txt`.

## Recommended design direction

Integrate schema checks into `mdb validate`, because schema validation is an
integrity check and the PO explicitly wants more determinism. A separate
operation may be considered in a future backlog item, but it is not required for
this item.

The likely contract is:

- `mdb validate --root <root>` validates schema references found in section
  metadata.
- Schema validation failures are reported as validation errors.
- `mdb validate --json` includes enough structured fields for agents to locate
  the failing section, schema, schema path, and metadata path.

## Out of scope

- Inventing a custom schema language.
- Validating Markdown body content.
- Requiring every section to declare a schema.
- Defining a global repository schema.
- Breaking existing repositories that do not use the `schema` attribute.
- Requiring network access for local-only validation workflows.
- Resolving schemas from web URIs. This is deferred to B-034.
- Changing API or CLI contracts without versioned or additive documentation.

## Criteria of acceptance

- [x] A section without a `schema` attribute keeps current validation behavior.
- [x] A section with a local JSON Schema reference under `scrum/schema/` is
      validated against that schema.
- [x] A section with a local YAML schema reference under `scrum/schema/` is
      validated when the YAML file encodes JSON Schema.
- [x] A valid section metadata block passes schema validation.
- [x] An invalid section metadata block fails schema validation with a clear
      error.
- [x] Error output includes the section URI, schema reference, failing metadata
      path, and validation detail.
- [x] Missing local schema files produce deterministic validation errors.
- [x] Invalid schema documents produce deterministic validation errors.
- [x] Web URI schema references are not resolved in this item and return a
      deterministic validation error or documented unsupported result.
- [x] `mdb validate --json` remains backward compatible or is extended
      additively with documented fields.
- [x] Automated tests cover local JSON schema, local YAML schema, valid
      metadata, invalid metadata, missing schema, invalid schema, and sections
      without schemas.
- [x] Documentation explains where to store local schemas, how `schema`
      references are resolved, that schemas are per-section only, and how
      validation errors are reported.

## Risk assessment

- Implementation risk: high, because schema resolution must handle local files,
  JSON/YAML parsing, and deterministic error reporting.
- Contract risk: medium, because `mdb validate --json` may need additive fields
  while preserving existing consumers.
- Dependency risk: high, because JSON Schema validation may require adding or
  selecting a validation library and documenting its supported draft.
- Network risk: low in this item, because web URI schemas are explicitly
  deferred to B-034.
- Test risk: medium, because fixtures must cover valid and invalid schemas
  without becoming brittle.

## Notes

- The PO asked for schema references through a `schema` metadata attribute.
- Local schemas should live under `scrum/schema/` by convention.
- No global schema should be introduced; validation remains opt-in per section.
- Web URI schemas are intentionally deferred to preserve local deterministic
  validation.
- This item intentionally keeps schema validation opt-in per section.
